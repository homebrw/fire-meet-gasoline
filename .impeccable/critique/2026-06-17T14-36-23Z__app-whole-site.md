## Overall Impression

**What works:** Clean, calm interface built on a well-documented design system. Color semantics are intentional and carry meaning. The layout is responsive and purposeful. French localization is complete and natural.

**What doesn't work:** Help and guidance are nearly invisible. Error states, empty states, and edge cases lack design. Power users have no documented shortcuts or efficiency paths.

**Biggest opportunity:** Add error, empty, and loading states—and pair them with contextual help. These are the moments users need reassurance in a high-stakes coordination tool.

## What's Working

1. **Semantic color system**: Person colors (blue/pink/cyan) and status colors (green/orange/gray) are applied consistently and always paired with text or icons. Users can build mental models quickly. Light/dark mode is equally refined.

2. **Responsive layout:** Sidebar on desktop, bottom nav on mobile—a familiar and functional pattern. The `pb-safe` utility shows attention to iPhone notch/safe area. No forced breakpoints; responsive behavior is structural.

3. **Information hierarchy and calm:** Dashboard prioritizes one piece of information at a time (today's status, next available slot, upcoming transitions). No visual noise. The card-based layout with generous padding creates breathing room.

## Priority Issues

### [P1] Missing Error States and Empty States

**What:** Settings pages and data-driven views (calendar, week, events) have no visible error boundaries, loading states, or empty state messaging.

**Why it matters:** Co-parents rely on this app for custody coordination. A blank screen or generic "no data" message creates uncertainty. They need confidence the system is working, or they'll abandon it.

**Fix:** Design and implement:
- Empty states with actionable next steps ("No custody rules created yet. [Add Rule]")
- Loading skeleton states for async data (card skeletons while calendar loads)
- Error states with recovery suggestions ("Failed to save rule. Try again or contact support.")
- Loading indicators for multi-second operations

**Suggested command:** `/impeccable polish` (treat as part of production hardening)

---

### [P1] No Contextual Help or Guidance

**What:** The interface has no tooltips, inline hints, contextual help, or onboarding for first-time users.

**Why it matters:** Custody rules, recurrence patterns, and transition management are complex. A first-time user will hesitate at every settings page. First-impression failure = churn.

**Fix:** Add:
- Inline help text below each form field in settings
- Tooltips on unfamiliar terms (e.g., "Recurrence" → "Pattern for repeating custody schedules")
- Empty-state guided tours for settings sections
- A 3-step onboarding flow on first login (add people, create first custody rule, preview calendar)

**Suggested command:** `/impeccable onboard` (design first-run and empty-state flows)

---

### [P2] Error Prevention Weak for Destructive Actions

**What:** No confirmation dialogs or reversibility documented for deletes (rules, events, exceptions).

**Why it matters:** Deleting a custody rule by mistake could disrupt the entire family's schedule. Heuristic #5 (Error Prevention) rates 2/4 because prevention is not visible in the code.

**Fix:** Add:
- Confirmation dialogs for delete actions with clear consequences ("Deleting this rule will remove all custody periods after [date]")
- Soft deletes (archive instead of hard delete) for auditable history
- Undo/recovery window (5-minute recovery on destructive actions)

**Suggested command:** `/impeccable harden` (add confirmation flows and reversibility)

---

### [P2] No Power-User Efficiency Paths

**What:** Navigation requires clicking through pages. No keyboard shortcuts, bulk actions, or power-user workflows.

**Why it matters:** Co-parents managing families with multiple kids and complex schedules will want to batch-edit events or create rules faster. No shortcuts = frustration for power users (Heuristic #7 scores 2/4).

**Fix:** Add:
- Keyboard shortcuts for common actions (⌘/Ctrl+K for command palette, `d` for today, `c` for calendar, `s` for settings)
- Bulk event edit (select multiple, apply action)
- Quick-add buttons on dashboard (e.g., "Quick Add Event")
- Customizable shortcuts for power users

**Suggested command:** `/impeccable shape` (plan efficiency improvements before implementation)

---

### [P3] Limited Visual Feedback for Async Operations

**What:** Form submissions, data saves, and page transitions are not clearly signaled with loading states or success confirmations.

**Why it matters:** Users may double-click submit or leave a page thinking the action failed. In a coordination tool, unclear feedback creates doubt.

**Fix:** Add:
- Button loading states (spinner or disabled state during submit)
- Toast notifications for success/error (e.g., "Rule saved" with green checkmark)
- Page-level loading skeleton when navigating between sections
- Optimistic UI updates (show new event immediately, confirm on server response)

**Suggested command:** `/impeccable polish` (add feedback animations and toasts)

---

### [P3] Contrast Verification Needed on Colored Backgrounds

**What:** Badge combinations like "outline" variant text on person-colored backgrounds have not been tested for WCAG AA (4.5:1) contrast at runtime.

**Why it matters:** DESIGN.md specifies 4.5:1 contrast for body text. Some badge/background combinations may not hit this under all conditions (e.g., light cyan background with muted text).

**Fix:** Audit and fix:
- Test all badge + background color combinations for 4.5:1 contrast
- Adjust badge text colors or backgrounds if contrast fails
- Use the detector with browser visualization to catch CSS-rendered contrast issues
- Verify placeholder text contrast (another 4.5:1 requirement often missed)

**Suggested command:** `/impeccable audit` (technical a11y, contrast, performance check)

## Minor Observations

1. **Navigation consistency:** Bottom nav icons are labeled (good for Jordan), but sidebar links are text + icon. Label visibility across screen sizes should be tested.

2. **Card shadows:** The design system specifies "Shadow-MD" but actual implementation uses Tailwind defaults. Verify shadow tokens match DESIGN.md.

3. **Form labels and focus:** Input fields use CSS custom properties, but form labels should be tested for proper `<label>` association and screen-reader pairing.

4. **Responsive type scaling:** Headings use fixed sizes (`text-2xl`, `text-base`). Per the product register, fixed rem scale is appropriate, but verify max line lengths stay within 65–75ch on all breakpoints.

5. **Language consistency:** French is well done, but verify all error messages, empty states, and help text follow the same tone and formality level.

## Questions to Consider

- **"How do first-time users learn to create a custody rule?"** Right now, settings has no onboarding. Should there be a guided tour or an empty-state walkthrough?

- **"What happens when custody data fails to load?"** Is there a fallback UI, or does the page blank out? This is a critical failure mode.

- **"Are power users (coordinating 3+ kids, multiple people) slowed down by the interface?"** No shortcuts means every action is a full navigation cycle.

- **"Does the left-border color actually help users scan the dashboard faster, or is it just decoration?"** It's documented as intentional, but user testing would confirm the value.

- **"What's the recovery path if a user deletes a custody rule by mistake?"** Soft delete, undo window, or just gone? This is high-stakes UX.
